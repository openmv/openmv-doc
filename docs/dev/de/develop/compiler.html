Der Compiler¶

Der Kompilierungsprozess in MicroPython umfasst die folgenden Schritte:

  • Der Lexer wandelt den Textstrom, aus dem ein MicroPython-Programm besteht, in Tokens um.

  • Der Parser wandelt die Tokens anschließend in eine abstrakte Syntax (Parse-Baum) um.

  • Anschließend wird auf Basis des Parse-Baums Bytecode oder nativer Code erzeugt.

Für die Zwecke dieser Erörterung fügen wir ein einfaches Sprachmerkmal add1 hinzu, das in Python wie folgt verwendet werden kann:

>>> add1 3
4
>>>

Die add1-Anweisung nimmt eine Ganzzahl als Argument entgegen und addiert 1 dazu.

Eine Grammatikregel hinzufügen¶

Die Grammatik von MicroPython basiert auf der CPython-Grammatik und ist in py/grammar.h definiert. Diese Grammatik wird zum Parsen von MicroPython-Quelldateien verwendet.

Es gibt zwei Makros, die Sie zum Definieren einer Grammatikregel kennen müssen: DEF_RULE und DEF_RULE_NC. DEF_RULE ermöglicht es Ihnen, eine Regel mit einer zugehörigen Compile-Funktion zu definieren, während DEF_RULE_NC keine Compile-Funktion (NC) besitzt.

Eine einfache Grammatikdefinition mit einer Compile-Funktion für unsere neue add1-Anweisung sieht wie folgt aus:

DEF_RULE(add1_stmt, c(add1_stmt), and(2), tok(KW_ADD1), rule(testlist))

Das zweite Argument c(add1_stmt) ist die zugehörige Compile-Funktion, die in py/compile.c implementiert werden sollte, um diese Regel in ausführbaren Code umzuwandeln.

Das dritte erforderliche Argument kann or oder and sein. Es gibt die Anzahl der mit einer Anweisung verknüpften Knoten an. In diesem Fall ähnelt unsere add1-Anweisung beispielsweise dem ADD1-Befehl in Assemblersprache. Sie nimmt ein numerisches Argument entgegen. Daher sind mit der add1_stmt zwei Knoten verknüpft. Ein Knoten steht für die Anweisung selbst, also das Literal add1, das KW_ADD1 entspricht, und der andere für ihr Argument, eine testlist-Regel, die die Ausdrucksregel der obersten Ebene ist.

Bemerkung

Die add1-Regel ist hier nur ein Beispiel und nicht Teil der Standardgrammatik von MicroPython.

Das vierte Argument in diesem Beispiel ist das mit der Regel verknüpfte Token, KW_ADD1. Dieses Token sollte im Lexer durch Bearbeiten von py/lexer.h definiert werden.

Dieselbe Regel ohne Compile-Funktion zu definieren, gelingt durch Verwendung des Makros DEF_RULE_NC und Weglassen des Compile-Funktionsarguments:

DEF_RULE_NC(add1_stmt, and(2), tok(KW_ADD1), rule(testlist))

Die übrigen Argumente behalten dieselbe Bedeutung. Eine Regel ohne Compile-Funktion muss explizit von allen Regeln behandelt werden, die diese Regel als Knoten enthalten können. Solche NC-Regeln werden üblicherweise verwendet, um Teilbereiche einer komplizierten Grammatikstruktur auszudrücken, die sich nicht in einer einzigen Regel ausdrücken lassen.

Bemerkung

Die Makros DEF_RULE und DEF_RULE_NC nehmen weitere Argumente entgegen. Für ein tiefergehendes Verständnis der unterstützten Parameter siehe py/grammar.h.

Ein lexikalisches Token hinzufügen¶

Jede in der Grammatik definierte Regel sollte ein zugehöriges Token besitzen, das in py/lexer.h definiert ist. Fügen Sie dieses Token hinzu, indem Sie das Enum _mp_token_kind_t bearbeiten:

typedef enum _mp_token_kind_t {
    ...
    MP_TOKEN_KW_OR,
    MP_TOKEN_KW_PASS,
    MP_TOKEN_KW_RAISE,
    MP_TOKEN_KW_RETURN,
    MP_TOKEN_KW_TRY,
    MP_TOKEN_KW_WHILE,
    MP_TOKEN_KW_WITH,
    MP_TOKEN_KW_YIELD,
    MP_TOKEN_KW_ADD1,
    ...
} mp_token_kind_t;

Bearbeiten Sie dann auch py/lexer.c, um den neuen Schlüsselwort-Literaltext hinzuzufügen:

static const char *const tok_kw[] = {
    ...
    "or",
    "pass",
    "raise",
    "return",
    "try",
    "while",
    "with",
    "yield",
    "add1",
    ...
};

Beachten Sie, dass das Schlüsselwort je nach gewünschtem Namen benannt wird. Halten Sie der Konsistenz halber den Benennungsstandard entsprechend ein.

Bemerkung

Die Reihenfolge dieser Schlüsselwörter in py/lexer.c muss mit der Reihenfolge der Tokens im Enum übereinstimmen, das in py/lexer.h definiert ist.

Parsen¶

In der Parsing-Phase nimmt der Parser die vom Lexer erzeugten Tokens und wandelt sie in einen abstrakten Syntaxbaum (AST) bzw. Parse-Baum um. Die Implementierung des Parsers ist in py/parse.c definiert.

Der Parser pflegt außerdem eine Konstantentabelle zur Verwendung in verschiedenen Aspekten des Parsens, ähnlich wie es eine Symboltabelle tut.

Während dieser Phase werden mehrere Optimierungen wie Constant Folding bei Ganzzahlen für die meisten Operationen, z. B. logische, binäre, unäre usw., sowie optimierende Verbesserungen bei Klammern um Ausdrücke durchgeführt, zusammen mit einigen Optimierungen bei Strings.

Es ist erwähnenswert, dass Docstrings verworfen werden und für den Compiler nicht zugänglich sind. Selbst Optimierungen wie String Interning werden nicht auf Docstrings angewendet.

Compiler-Durchläufe¶

Wie viele Compiler kompiliert MicroPython sämtlichen Code zu MicroPython-Bytecode oder nativem Code. Die Funktionalität, die dies erreicht, ist in py/compile.c implementiert. Die wichtigste Methode, die Sie kennen sollten, ist diese:

mp_obj_t mp_compile(mp_parse_tree_t *parse_tree, qstr source_file, bool is_repl) {
    // Create a context for this module, and set its globals dict.
    mp_module_context_t *context = m_new_obj(mp_module_context_t);
    context->module.globals = mp_globals_get();

    // Compile the input parse_tree to a raw-code structure.
    mp_compiled_module_t cm;
    cm.context = context;
    mp_compile_to_raw_code(parse_tree, source_file, is_repl, &cm);

    // Create and return a function object that executes the outer module.
    return mp_make_function_from_proto_fun(cm.rc, cm.context, NULL);
}

Der Compiler kompiliert den Code in vier Durchläufen: Geltungsbereich (Scope), Stack-Größe, Code-Größe und Emittieren. Jeder Durchlauf führt denselben C-Code über dieselbe AST-Datenstruktur aus, wobei jedes Mal auf Basis der Ergebnisse des vorherigen Durchlaufs unterschiedliche Dinge berechnet werden.

Erster Durchlauf¶

Im ersten Durchlauf lernt der Compiler die bekannten Bezeichner (Variablen) und deren Geltungsbereich kennen, sei es global, lokal, geschlossen über (closed over) usw. Im selben Durchlauf berechnet der Emitter (Bytecode oder nativer Code) außerdem die Anzahl der für den emittierten Code benötigten Labels.

// Compile pass 1.
comp->emit = emit_bc;
comp->emit_method_table = &emit_bc_method_table;

uint max_num_labels = 0;
for (scope_t *s = comp->scope_head; s != NULL && comp->compile_error == MP_OBJ_NULL; s = s->next) {
    if (s->emit_options == MP_EMIT_OPT_ASM) {
        compile_scope_inline_asm(comp, s, MP_PASS_SCOPE);
    } else {
        compile_scope(comp, s, MP_PASS_SCOPE);

        // Check if any implicitly declared variables should be closed over.
        for (size_t i = 0; i < s->id_info_len; ++i) {
            id_info_t *id = &s->id_info[i];
            if (id->kind == ID_INFO_KIND_GLOBAL_IMPLICIT) {
                scope_check_to_close_over(s, id);
            }
        }
    }
    ...
}

Zweiter und dritter Durchlauf¶

Der zweite und dritte Durchlauf umfassen die Berechnung der Python-Stack-Größe und der Code-Größe für den Bytecode oder nativen Code. Nach dem dritten Durchlauf darf sich die Code-Größe nicht mehr ändern, andernfalls werden Sprung-Labels falsch.

for (scope_t *s = comp->scope_head; s != NULL && comp->compile_error == MP_OBJ_NULL; s = s->next) {
    ...

    // Pass 2: Compute the Python stack size.
    compile_scope(comp, s, MP_PASS_STACK_SIZE);

    // Pass 3: Compute the code size.
    if (comp->compile_error == MP_OBJ_NULL) {
        compile_scope(comp, s, MP_PASS_CODE_SIZE);
    }

    ...
}

Kurz vor dem zweiten Durchlauf erfolgt eine Auswahl des zu emittierenden Code-Typs, der entweder nativ oder Bytecode sein kann.

// Choose the emitter type.
switch (s->emit_options) {
    case MP_EMIT_OPT_NATIVE_PYTHON:
    case MP_EMIT_OPT_VIPER:
        if (emit_native == NULL) {
            emit_native = NATIVE_EMITTER(new)(&comp->compile_error, &comp->next_label, max_num_labels);
        }
        comp->emit_method_table = NATIVE_EMITTER_TABLE;
        comp->emit = emit_native;
        break;

    default:
        comp->emit = emit_bc;
        comp->emit_method_table = &emit_bc_method_table;
        break;
}

Die Bytecode-Option ist die Standardeinstellung, aber etwas Besonderes ist bei der Option für nativen Code zu beachten: Es gibt eine weitere Option über VIPER. Weitere Einzelheiten zu Viper-Annotationen finden Sie im Abschnitt Nativen Code emittieren.

Es gibt außerdem Unterstützung für Inline-Assembler-Code, bei dem Assembler-Anweisungen als Python-Funktionsaufrufe geschrieben, aber direkt als entsprechender Maschinencode emittiert werden. Dieser Assembler hat nur drei Durchläufe (Scope, Code-Größe, Emittieren) und verwendet eine andere Implementierung, nicht die Funktion compile_scope. Weitere Einzelheiten finden Sie in der Referenz zum Inline-Assembler.

Vierter Durchlauf¶

Der vierte Durchlauf emittiert den endgültigen, ausführbaren Code, entweder Bytecode in der virtuellen Maschine oder nativen Code direkt durch die CPU.

for (scope_t *s = comp->scope_head; s != NULL && comp->compile_error == MP_OBJ_NULL; s = s->next) {
    ...

    // Pass 4: Emit the compiled bytecode or native code.
    if (comp->compile_error == MP_OBJ_NULL) {
        compile_scope(comp, s, MP_PASS_EMIT);
    }
}

Bytecode emittieren¶

Anweisungen im Python-Code entsprechen üblicherweise emittiertem Bytecode, beispielsweise erzeugt a + b „push a“, dann „push b“ und dann „binary op add“. Manche Anweisungen emittieren nichts, sondern beeinflussen stattdessen andere Dinge wie den Geltungsbereich von Variablen, beispielsweise global a.

Die Implementierung einer Funktion, die Bytecode emittiert, sieht in etwa so aus:

void mp_emit_bc_unary_op(emit_t *emit, mp_unary_op_t op) {
    emit_write_bytecode_byte(emit, 0, MP_BC_UNARY_OP_MULTI + op);
}

Wir verwenden hier die Ausdrücke der unären Operatoren als Beispiel, aber die Implementierungsdetails sind für andere Anweisungen/Ausdrücke ähnlich. Die Methode emit_write_bytecode_byte() ist ein Wrapper um die Hauptfunktion emit_get_cur_to_write_bytecode(), die alle Funktionen aufrufen müssen, um Bytecode zu emittieren.

Nativen Code emittieren¶

Ähnlich wie Bytecode erzeugt wird, sollte es für jede Code-Anweisung eine entsprechende Funktion in py/emitnative.c geben:

static void emit_native_unary_op(emit_t *emit, mp_unary_op_t op) {
     vtype_kind_t vtype;
     emit_pre_pop_reg(emit, &vtype, REG_ARG_2);
     if (vtype == VTYPE_PYOBJ) {
         emit_call_with_imm_arg(emit, MP_F_UNARY_OP, op, REG_ARG_1);
         emit_post_push_reg(emit, VTYPE_PYOBJ, REG_RET);
     } else {
         adjust_stack(emit, 1);
         EMIT_NATIVE_VIPER_TYPE_ERROR(emit,
             MP_ERROR_TEXT("unary op %q not implemented"), mp_unary_op_method_name[op]);
     }
}

Der Unterschied besteht hier darin, dass wir die Viper-Typisierung behandeln müssen. Viper-Annotationen ermöglichen es uns, mehr als einen Variablentyp zu behandeln. Standardmäßig sind alle Variablen Python-Objekte, aber mit Viper kann eine Variable auch als maschinentypisierte Variable deklariert werden, etwa als native Ganzzahl oder als Zeiger. Viper kann als eine Obermenge von Python betrachtet werden, bei der normale Python-Objekte wie gewohnt behandelt werden, während native Maschinenvariablen auf optimierte Weise behandelt werden, indem direkte Maschinenbefehle für die Operationen verwendet werden. Die Viper-Typisierung kann die Python-Äquivalenz brechen, da beispielsweise Ganzzahlen zu nativen Ganzzahlen werden und überlaufen können (im Gegensatz zu Python-Ganzzahlen, die sich automatisch auf beliebige Genauigkeit erweitern).
